## Overview

The database has six core tables:

1. `users` — authentication and account management
2. `golfer_profiles` — the My Golf profile for each user
3. `sessions` — each range or simulator session
4. `shots` — individual shot records within a session
5. `session_summaries` — post-session summaries generated by Whisper
6. `programme_progress` — mastery tracking per user per programme (Birdie tier)

## Input Validation Layer

A lightweight relevance check that runs before any shot is saved to the database. Prevents junk data from polluting Whisper's coaching context.

### How It Works

```
Golfer speaks → Transcript captured → Validation check → Decision
                                              ↓
                              [Golf relevant?] → YES → Save to database, send to Whisper
                                              → NO  → Whisper responds but shot NOT saved
```

### Validation Rules

| Check | Method | Action if Failed |
|---|---|---|
| Golf relevance | Keyword presence + Claude mini-call scoring 0–1 | Score below 0.3 → discard, Whisper redirects |
| Minimum length | Under 3 words | Prompt golfer to say more |
| Club mentioned or inferable | Club name or context present | Ask clarifying question before saving |
| Miss type present or inferable | Outcome described | Ask clarifying question before saving |
| Duplicate detection | Same transcript within 30 seconds | Flag as accidental double-tap, don't save twice |

### Junk Examples That Get Discarded

- "Hello Whisper I love Latina girls" → score: 0.0 → discarded
- "Testing testing 1 2 3" → score: 0.1 → discarded
- "What's the weather like" → score: 0.05 → discarded
- "I hit it" → score: 0.4 → saved but flagged as low quality (no club, no miss type)

### Implementation Notes

- Validation runs server-side in `server.js` before the shot is passed to Whisper
- Low-cost mini Claude call (~0.001 cents) — worth it for data integrity
- Whisper always responds to the golfer regardless of validation outcome — the golfer never knows their input was discarded
- Discarded inputs are logged to a separate `discarded_inputs` table for monitoring

### discarded_inputs Table

| Column | Type | Notes |

## Future: Data Hygiene Agent (Phase 3+)

A background AI agent that runs nightly to clean, standardise and compress the database as user volume grows. Not needed at MVP — activate when user count exceeds 100.

### Agent Tasks

| Task | Frequency | Description |
|---|---|---|
| **Miss type standardisation** | Nightly | Maps informal language to standard miss types. "Duffed it" → `fat`, "ballooned it" → `balloon`, "shanked it horrifically" → `shank` |
| **Low confidence flagging** | Nightly | Tags shots where Whisper's inferred cause confidence is low — prevents weak data influencing pattern detection |
| **Anomaly detection** | Nightly | Flags suspicious patterns: 200 shots in 10 minutes, same miss type 50 times in a row, impossible club sequences |
| **Context compression** | Weekly | Summarises old shot history into compressed pattern summaries — keeps Whisper's context window lean as history grows |
| **Profile staleness check** | Weekly | Flags users whose handicap or profile hasn't been updated in 90 days — prompts them to review |
| **Dead session cleanup** | Nightly | Closes sessions that were started but never ended (app crashed, user forgot) |

### Agent Architecture

```
Scheduled job (Railway cron) → runs at 2am daily
  → Query shots logged in last 24 hours
  → Run standardisation pass
  → Run anomaly detection
  → Flag low confidence inferred causes
  → Log agent actions to agent_log table
  → Send daily digest to founder (email or Slack)
```

### agent_log Table

| Column | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `run_at` | TIMESTAMP | When agent ran |
| `shots_reviewed` | INTEGER | |
| `shots_standardised` | INTEGER | |
| `anomalies_flagged` | INTEGER | |
| `sessions_closed` | INTEGER | |
| `notes` | TEXT | Summary of what was changed |

---



**Why UUIDs not integers?**
UUIDs are safer for user-facing IDs and allow distributed generation without collision risk.

**Why JSONB for shot_context?**
The Bag feature's context fields will evolve — JSONB allows flexibility without schema migrations.

**Why TEXT[] arrays for tags and patterns?**
Postgres native arrays are clean for this use case. If querying complexity grows, migrate to a junction table.

**Why store user_id on shots directly?**
Denormalisation for query performance — fetching all of a user's shots across sessions is a common operation for Whisper's context building.

**Why sentiment on the shot level?**
Sentiment needs to be tracked per shot to detect in-session emotional shifts, not just overall session mood.

## Context Window Strategy

Whisper receives shot history as context in each API call. To manage token usage:

- **MVP:** Last 20 shots from current session + last 3 session summaries
- **Birdie:** Last 50 shots from current session + last 10 session summaries + programme progress notes
- **Eagle (future):** Full rolling window with intelligent summarisation

---

## Data Retention Notes

- Shot data is the core product asset — never delete
- If a user deletes their account, anonymise rather than delete shot data (aggregate analytics value)
- Session summaries should be retained indefinitely — they are the product the user sees
